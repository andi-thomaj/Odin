# Backend (`Odin/`) — Claude Code rules

This file is loaded in addition to the root `CLAUDE.md` when working anywhere under `Odin/`.

## Auth — Auth0 JWT, DB-sourced roles, `email_verified` resolution

JWT bearer is configured in [Odin.Api/Program.cs](Odin.Api/Program.cs) (`Jwt:Authority` / `Jwt:Audience`).
After auth, [RoleEnrichmentMiddleware](Odin.Api/Middleware/RoleEnrichmentMiddleware.cs) attaches an
`app_role` claim from the `application_users` table (roles are **DB-sourced on purpose** so promotions
take effect immediately) and an `app_email_verified` claim. The middleware is fully bypassed under the
`Testing` environment — [TestAuthHandler](Odin.Api/Authentication/TestAuthHandler.cs) supplies claims
from `X-Test-*` headers instead.

**`email_verified` resolution + the `/userinfo` optimization.** Auth0 API access tokens usually omit
`email_verified`, so the middleware falls back to calling Auth0 `/userinfo` (cached per-token: 15 min
verified / 2 min unverified). To remove that network call from the hot path, add a **post-login Auth0
Action** that stamps a namespaced claim onto the **access token**, then set `Jwt:EmailVerifiedClaim` to
that exact claim type:

```js
// Auth0 → Actions → Login flow
exports.onExecutePostLogin = async (event, api) => {
  const ns = "https://odin.ancestrify.io/"; // any namespace you control; must match Jwt:EmailVerifiedClaim
  if (event.authorization) {
    api.accessToken.setCustomClaim(`${ns}email_verified`, event.user.email_verified === true);
  }
};
```

Set `Jwt:EmailVerifiedClaim` (appsettings / `Jwt__EmailVerifiedClaim` env var) to
`https://odin.ancestrify.io/email_verified`. [Auth0EmailVerifiedClaims.GetJwtEmailVerifiedBoolean](Odin.Api/Authentication/Auth0EmailVerifiedClaims.cs)
reads it first; the `/userinfo` fallback stays for tokens issued before the Action, so rollout is
zero-downtime and the fallback can be retired once all live tokens carry the claim. Leaving the key
empty (the default) preserves the original `/userinfo` behaviour.

The per-request role lookup is cached via [UserRoleCacheKeys](Odin.Api/Authentication/UserRoleCacheKeys.cs)
and invalidated on every write to `User.Role` (see [UserService](Odin.Api/Endpoints/UserManagement/UserService.cs)
`UpdateUserRoleAsync` / `DeleteUserAsync`) so promotions still apply immediately.

## Single application — identity keyed on the Auth0 sub (multi-app isolation REMOVED)

This backend serves **one** application. The web (odin-charmander) and the iOS app are the same product to the
backend, so a user is **one account everywhere**: identity is `application_users.IdentityId` (the Auth0 sub),
which is **globally unique**. Reference/seed data is shared.

The former **multi-app data-isolation** capability (a separate account + data silo per `X-App` value) was
**removed** — migration `RemoveMultiAppIsolation` drops the `App` column from every user-owned table, drops the
`applications` table, and re-points the indexes (e.g. `application_users` unique on `IdentityId` alone). **Do not
reintroduce it**: there is no `X-App` header, no `AppResolutionMiddleware`, no `IAppContext`/`RequestAppContext`,
no `IAppScoped`, no `App` column, no `applications` registry/seeder. Frontends no longer send `X-App` / `?app=`.

- The auth hot path keys on the sub alone: provisioning inserts `User { IdentityId }`, role-lookup cache key is
  `UserRoleCacheKeys.ForIdentity(identityId)`.
- The **only** remaining global query filter is `RawGeneticFile`'s soft-delete (`!IsDeleted`) in
  [ApplicationDbContext](Odin.Api/Data/ApplicationDbContext.cs). Background jobs that must see soft-deleted files
  (e.g. `MergeJob`) call `IgnoreQueryFilters()` purely to bypass that — there is no app dimension anymore.
- **First admin** is still promoted via a one-time DB edit:
  `UPDATE application_users SET role='Admin' WHERE identity_id=…`.

## Caching — in-process `IMemoryCache`, single instance, invalidate on write

The API runs as a **single instance and is not designed to scale horizontally**, so caching is
plain in-process `IMemoryCache` (registered in [Odin.Api/Program.cs](Odin.Api/Program.cs)) — no
Redis / `IDistributedCache`. Follow the established pattern when caching read-mostly reference data:
`TryGetValue` → query on miss → `Set` with an `AbsoluteExpirationRelativeToNow` TTL safety net;
**skip the cache under the `Testing` environment** (`IHostEnvironment.IsEnvironment("Testing")`, so
integration tests read fresh); and **invalidate on write** via `cache.Remove(key)` in the
create/update/delete/import paths. Centralise keys in a small static class (e.g.
`OrderResultCacheKeys`, `G25SampleCacheKeys`) rather than inlining strings, and keep
[BackendCacheMaintenanceService](Odin.Api/Services/BackendCacheMaintenanceService.cs)'s "what gets
cleared" doc-comment current. Reference implementations: `EraService` (`AllEras`),
`G25CalculationService` (per-era distance samples), `PopulationService`/`EthnicityService`
(invalidation). Admin can flush everything via `POST /v1/api/admin/cache/clear`.

## API versioning — `/v1` today, side-by-side `/v2` for breaking changes

All business endpoints are mounted under `/v1` ([Odin.Api/Program.cs](Odin.Api/Program.cs) → `var v1 = app.MapGroup("/v1");`). SignalR hubs (`/hubs/...`) and infrastructure routes (`/health`, `/jobs`, `/swagger`) stay at the root by convention — they're not part of the versioned API surface.

When a request/response contract has to change in a **breaking** way (renamed field, semantic change, removed field), add a `/v2` group alongside `/v1` rather than mutating the existing endpoint. Same handler can serve both by accepting a wider DTO and projecting per version, or each version can have its own endpoint methods — pick whichever keeps the per-endpoint code clearest. `/v1` stays alive until the FE schema regenerates, the FE migrates, and a deprecation window passes.

Non-breaking changes (new field on a response, new optional field on a request) can ship in place on `/v1` — they're already non-breaking by construction. Don't pre-emptively v2 for these.

## Integration test database — Testcontainers Postgres

`Odin.Api.IntegrationTests` boots a disposable **PostgreSQL 16 container** (via `Testcontainers.PostgreSql`) on every test run. Running the suite locally requires Docker. CI uses GitHub-hosted `ubuntu-latest`, where Docker is preinstalled — see [.github/workflows/backend-tests.yml](.github/workflows/backend-tests.yml), which runs both `Odin.Api.Tests` (unit) and `Odin.Api.IntegrationTests` as parallel jobs on every PR and on pushes to `master` / `development`.

To target an **external** Postgres instead (e.g. a locally running dev database), set `ConnectionStrings__DefaultConnection` in the environment before invoking `dotnet test`. The factory skips the container when the env var is non-empty and runs Respawn against the supplied database. Leave it unset (or empty) to use the container.

Do not point integration tests at a database that holds data you care about — Respawn wipes every public table between tests.

## Integration tests — do not run the full suite after every code change

**Never** run the entire `Odin.Api.IntegrationTests` project as the default "verify my edit" step after local code changes. The suite is **slow** (Postgres container boot, heavy seeding, many HTTP flows) and routinely takes many minutes.

**Do** run only the tests you judge **impacted** by what you changed — for example a single class, namespace, or feature area — using `--filter`:

```bash
cd Odin
dotnet test Odin.Api.IntegrationTests/Odin.Api.IntegrationTests.csproj --filter "FullyQualifiedName~EraEndpointsTests"
```

Other examples: `FullyQualifiedName~GeneticInspectionEndpointsTests`, `FullyQualifiedName~OrderEndpointsTests`.

Reserve **full** `dotnet test` on `Odin.Api.IntegrationTests` for CI, pre-merge checks, release validation, or when your change touches **shared test infrastructure** (e.g. `CustomWebApplicationFactory`, collection fixtures, `IntegrationTestBase`, global middleware/DI used by most tests).

Unit tests (`Odin.Api.Tests`) are fast; prefer them when they cover the change. Run targeted integration tests when the behavior under test is HTTP/DB/integration-specific.

## After changing a request or response contract — refresh the FE OpenAPI client

The frontend consumes BE types via [`odin-react/src/api/_generated/schema.d.ts`](../odin-react/src/api/_generated/schema.d.ts), produced by `npm run gen:api` from the Swashbuckle doc at `/swagger/v1/swagger.json`. **Whenever you add, rename, or change a field on any `*Contract.Request` / `*Contract.Response` (or add/remove an endpoint), the FE schema is stale until it's regenerated.**

After the BE change builds:

1. **Run the BE locally** so the OpenAPI doc updates: `cd Odin/Odin.Api && dotnet run` (listens on 5190).
2. **In a second terminal, from `odin-react/`:** `npm run gen:api` — overwrites `src/api/_generated/schema.d.ts`.
3. **Commit the regenerated `schema.d.ts` in the same commit/PR** as the BE contract change. CI should treat schema drift between BE and committed FE snapshot as a failure.

If you add a new endpoint, also remember to annotate the success response on the route registration so it appears in the OpenAPI doc:

```csharp
endpoints.MapGet("/foo", GetFoo)
    .RequireAuthorization("EmailVerified")
    .Produces<GetFooContract.Response>(StatusCodes.Status200OK);
```

Without `.Produces<T>()`, Swashbuckle emits an empty `content` for the 200 response and `gen:api` won't expose the type to the FE.

## Admin merge-panel restore — streamed multi-GB upload (raised limits)

The AADR `HO` merge panel is a pre-built upload (`v66_2M_aadr_PUB.{geno,snp,ind}`), not Poseidon-provisioned. `MergePanelAdminEndpoints` (`api/admin/merge-panel/*`, `AdminOnly`) proxies the tools-api `/v1/merge/panel/restore/*` flow: `GET status`, `POST upload` (one file at a time), `POST activate`. The frontend surfaces it as Admin → "Restore merge panel".

These files are **2–10 GB**, so the upload path deliberately departs from the global 50 MB limits:

- **Streamed, never buffered.** The browser sends each file as the **raw request body** (`application/octet-stream`); `ext`/`panel`/`sha256` ride as query params. The `upload` handler pipes `HttpContext.Request.Body` straight into `StreamContent` to the tools-api (`MergePipelineService.UploadPanelFileAsync`) — no `IFormFile`/multipart, no memory or temp-file spool.
- **Per-route Kestrel cap lifted.** The handler sets `IHttpMaxRequestBodySizeFeature.MaxRequestBodySize = null` (the global cap in `Program.cs` stays 50 MB for every other route). Must run before the body is read — hence reading `Request.Body` directly rather than model-binding a form.
- **Dedicated infinite-timeout client.** `MergePipelineService.PanelClientName` ("ToolsApiPanelRestore") has `Timeout = Timeout.InfiniteTimeSpan` so a slow multi-GB upload isn't killed by the 30-min merge-client timeout; it's bounded instead by the endpoint's **`PanelRestore`** request-timeout policy (2 h) + the request cancellation token.
- **Reverse proxy is the likely failure point.** Coolify/Traefik enforces its own request-body-size and read/idle timeouts; these must be raised for this route on the server (not in code), or a large upload 413s/times out before reaching .NET.

The 422 from activate carries a human-readable validation message (transposed `.geno`, `???` labels, truncated `.geno`); the FE shows it in a toast. Pass `force=true` to install despite "slow but usable" warnings.

## Panel sample labels — Scientist/Admin editor

`MergePanelLabelsEndpoints` (`api/merge-panel/labels`, `ScientistOrAdmin`) proxies the tools-api `/v1/merge/panel/ind*` routes so Scientists/Admins can correct the AADR panel's **population labels** (column 3 of the `.ind`) without re-uploading the multi-GB panel: `GET /` (list samples), `PUT /row` (edit one by row index), `POST /rename` (rename a label across all samples that have it). The frontend page is Tools → "Panel Labels" (`requireScientistOrAdminBeforeLoad`, `DataGridPro` inline edit + rename dialog).

The tools-api enforces the invariants (text-only on col 3, order-preserving — the `.geno` is keyed by row position; no whitespace in a label; atomic write + `*.ind.bak`); these endpoints just relay via `MergePipelineService` and map a 422 (validation) / 502 (tools-api unreachable) through the shared `Proxy(...)` helper. Adding methods to `IMergePipelineService` means the `StubMergeService` in `Odin.Api.Tests/.../MergeJobTests.cs` must implement them too, or the test project won't compile.

## Merge execution model — serialized, no auto-retry, admin-only retry

Merges run **one at a time** by **deliberate policy** (not a RAM constraint — the tools-api merges with Poseidon `trident` at ~1.3 GB; the old `mergeit` engine that needed ~25 GB was removed). The user wants merges kept strictly serialized; keep it that way.

- **Serialized — pinned to one at a time.** Merges run **strictly sequentially**: the dispatcher's in-flight cap (`MergeJob._maxInFlight`) and the Hangfire `merge`-queue `WorkerCount` are both **hardcoded to 1** in `Program.cs`. `MaxConcurrentMerges` is deliberately **not** bound from config (`Program.cs` wires `MergeJobOptions` to a literal `1`), so no appsettings/Coolify override can let two merges run concurrently. The `MergeJobOptions.MaxConcurrentMerges` setter survives only so `MergeJobTests` can exercise the dispatcher's admission arithmetic with other caps. **Do not raise these** — serialized-at-1 is a standing user preference, not a RAM limit.
- **⚠️ ALL Hangfire job-filter attributes MUST be on `IMergeJob` (the interface), not the concrete `MergeJob`.** Jobs are enqueued via `Enqueue<IMergeJob>(svc => svc.RunAsync(..))` / `AddOrUpdate<IMergeJob>(..)`, so Hangfire reads filter attributes off the **interface** method; the same attribute on the concrete method is silently ignored. This applies to `[Queue("merge")]` + `[AutomaticRetry(Attempts = 0)]` on `RunAsync` **and** `[DisableConcurrentExecution(60)]` on `DispatchPendingMergesAsync`. They were on the concrete `MergeJob` for a while, so merges actually ran on the multi-worker **`default`** queue (not the single-worker `merge` queue) AND got Hangfire's **default 10 retries** instead of `Attempts = 0`. This — combined with the invisibility bug below — let re-fetched merge jobs run **concurrently** (overlapping forges). The dispatcher had the same defect: `[DisableConcurrentExecution]` sat on the concrete method (so it was ignored), letting the every-minute recurring dispatch and the per-event dispatches (order creation, merge completion, requeue, stop) **race the count→admit step and over-admit past the cap of 1 — several merges showing "Merging" at once**. If you add any queue/retry/concurrency filter, put it on the interface. `MergeJobTests.HangfireFilters_LiveOnTheInterface_SoSerializationActuallyTakesEffect` reflection-asserts all of these so the regression can't return.
- **⚠️ Long merges + Hangfire invisibility timeout.** `UsePostgreSqlStorage` is configured with **`UseSlidingInvisibilityTimeout = true`** (+ a 2 h `InvisibilityTimeout` ceiling) in `Program.cs`. This is **load-bearing, not cosmetic**: the AADR merge is disk-bound and runs ~31 min on the shared host — longer than Hangfire.PostgreSql's **default fixed 30-min invisibility window**. Without the sliding timeout, the storage decided the still-running merge job was "lost" and **re-fetched + re-ran the same job every 30 min** — a perpetual loop (the merge never reached Succeeded/Failed; each re-run kicked off a fresh tools-api forge that again ran >30 min). The sliding timeout heartbeats the lease while a worker is genuinely processing, so a long merge runs exactly once. **Don't remove it**, and keep `ToolsApi:MergeTimeoutSeconds` (now 3600s) below that 2 h ceiling and above the real merge wall-time.
- **No automatic retries.** `RunAsync` is `[AutomaticRetry(Attempts = 0)]` (on the interface — see above); any failure (bad upload 400, panel unavailable 503, tool error 500, timeout) is recorded `MergeStatus.Failed` and does **not** rethrow — an immediate retry usually fails the same way and ties up the single worker; an admin re-runs it instead. `MergeStatus.Retrying` is now unused in the normal path; `MergeJobFailureStateFilter` still reconciles a hard-crashed (worker-died) job to Failed.
- **A failed merge must leave NO bundle on disk (bounded-disk invariant).** The tools-api itself is clean — all intermediates live in a per-merge temp workdir removed in `finally`, and the persistent `<merge_id>.tar.gz` is written only on the last step (a failure before that writes nothing; a partial tar is unlinked). The leak risk is .NET-side: a merge can fail while the tools-api **already wrote / is still writing** the bundle (a .NET client timeout doesn't cancel forge's uncancellable `run_in_threadpool`; also a lost response or a worker death). So on failure `RunCoreAsync` calls `DropAnyProducedBundleAsync` → `IMergePipelineService.CancelMergeAsync(mergeId)` (tools-api `/cancel` SIGKILLs a live forge **and** deletes any finished/partial bundle; uses `CancellationToken.None` so it runs even when the failure *was* a timeout/cancel; best-effort, never masks the original failure). **Backstop:** the weekly `CleanupOrphansAsync` also reclaims any `Failed` row that still carries a `MergeId` (the worker-died case where the inline cleanup never ran) via `TryDropFailedBundleAsync` — it deletes the bundle, keeps the `Failed` status + error, and nulls the `MergeId` (so it isn't revisited and an admin Retry mints a fresh id). Note neither the `Ready` orphan sweep nor an admin Retry would otherwise ever reclaim a `Failed` row's bundle, so this backstop is load-bearing for the 80 GB host. Covered by `MergeJobTests` (`RunAsync_MergeFailure_DropsAnyProducedBundle_SoNoTrashIsLeft`, `CleanupOrphans_ReclaimsOrphanedFailedBundle_*`).
- **Admin-initiated retry only.** `IMergeJob.RequeueAsync(rawGeneticFileId)` resets a non-running merge to `NotStarted` and dispatches; exposed as `POST api/admin/merge/{rawGeneticFileId}/retry` (`AdminOnly`) and surfaced as a **Retry** button on the Input page's results grid (admin-only, shown for `Failed` rows). FE: `src/api/merge-admin.ts` + `useRetryMerge`.
- **Admin Stop (cancel an in-progress merge).** `IMergeJob.StopAsync(rawGeneticFileId)` stops a `Queued`/`Converting`/`Merging` merge: it (1) deletes the running Hangfire `RunAsync` job (found via the monitoring API by inspection id — best-effort, so a monitoring hiccup can't block the stop) so the invocation's cancellation token fires and it never re-runs, (2) calls `IMergePipelineService.CancelMergeAsync(mergeId)` → tools-api `POST /v1/merge/{mergeId}/cancel`, which SIGKILLs the running tool subprocess (plink/trident/convertf) and drops any partial bundle, and (3) marks the file `Failed` ("Merge stopped by an administrator.") so an admin can Retry. Exposed as `POST api/admin/merge/{rawGeneticFileId}/stop` (`AdminOnly`), surfaced as a red **Stop** button on the Input grid for in-progress rows. FE: `mergeAdminApi.stop` + `useStopMerge`. (404 if the file is gone, 409 if no merge is in progress.)

NB: with trident a merge peaks at ~1.3 GB and never OOMs — no swap/host-resize needed. Merge *latency* (~31 min) is now bound by the host's slow disk, not RAM; that's an infra lever (faster disk), not a code one.

**Manual delete of a merged bundle.** `DELETE api/genetic-inspections/{id}/merged-data` (`ScientistOrAdmin`, in `GeneticInspectionEndpoints`) resolves the inspection's raw file and calls `IMergeJob.DeleteAsync` (removes the tools-api bundle, marks `Deleted`; idempotent). Surfaced on the Input results grid as a red delete button (shown for `Ready` rows), with the "Download merged data" icon shown green when a bundle exists (`mergeStatus === "Ready"`). FE: `geneticInspectionsApi.deleteMergedData` + `useDeleteMergedData`.

**Bulk delete of all merged bundles.** `DELETE api/genetic-inspections/merged-data` (`AdminOnly`, `strict` rate limit) calls `IMergeJob.DeleteAllReadyMergedDataAsync`, which deletes **every** `Ready` bundle on the tools-api volume right now (unconditional — unlike `CleanupOrphansAsync`, no retention/completed filter) and returns the count freed (`DeleteAllMergedDataContract.Response`). Each per-bundle delete is isolated so one failure doesn't abort the sweep. Surfaced on the Input results grid's toolbar as an admin-only "Delete all merged data" sweep button (badge = number of `Ready` bundles; disabled when zero). FE: `geneticInspectionsApi.deleteAllMergedData` + `useDeleteAllMergedData`. The literal `merged-data` route can't collide with `{id:int}/merged-data`.

## Public pre-launch waitlist subscribe endpoint

`POST /v1/api/public/subscribe` ([`SubscribeEndpoints`](Odin.Api/Endpoints/Subscribe/SubscribeEndpoints.cs), **`AllowAnonymous`**, `strict` rate limit) is the pre-launch waitlist signup the marketing site calls while self-service registration is disabled. It takes `{ email }`, validates it (`MailAddress.TryCreate`), and forwards to the Resend Audience via `IResendAudienceService.AddContactAsync` — **no DB entity**, Resend is the store of record for the waitlist. Resend/infra failures are **logged and masked** as `{ success: true }` so anonymous visitors never see a 500 (and we don't leak internals); only an invalid/empty email returns 400. Integration coverage: `SubscribeEndpointsTests` (the test host registers `NoOpResendAudienceService`, so no real Resend calls).

## G25 PCA population samples — per-individual reference cloud (derived from distance samples + coordinate files)

`g25_pca_populations_samples` is the PER-INDIVIDUAL PCA reference cloud (one row per sample), as opposed to `g25_distance_population_samples`, which holds population AVERAGES. Both entities share the SAME shape (`Label`, `Coordinates` text = 25 comma-separated PC values, `Ids` text, `ResearchLinks`) and reuse the SAME era catalog: a PCA sample's `G25DistanceEraId` points at `g25_distance_eras` (there is **no** separate PCA era table).

- **Shared derivation logic.** [`G25PcaSeedBuilder`](Odin.Api/Data/Seeders/G25PcaSeedBuilder.cs) (pure, no I/O/EF, in Odin.Api) owns the matching rules and the `EraFileNames` map, so the offline generator, the on-demand import test, and the unit test all agree. ANCIENT eras 1-5: each distance sample lists its member individual ids in `Ids`; look each up in that era's coordinate file (match the file's individual portion — text after the last `:` — exactly, else by boundary-prefix `id + "."` / `id + "_"`, e.g. `I14689` → `I14689.AG__BC_2555__Cov_50.21%`) and emit one PCA row per matched file row, `Label` = the distance sample's friendly population name. MODERN era 6: distance samples carry no ids, so ingest EVERY row of the Modern file as-is (`Label` = the file's population prefix). Ids that are coordinate-noise / a whole label leaked into the `Ids` column (contain `:`/`*` or are a decimal number) are dropped as dirty; ids absent from the era file are reported as unmatched — both for the report only. Filenames use a hyphen where the DB era names use an en-dash, but matching is by numeric era id, so no dash normalization is needed.
- **Source files are IN the repo.** The six per-era coordinate `.txt` files are committed under [`Data/SeedData/g25-coordinates/`](Odin.Api/Data/SeedData/g25-coordinates/) (raw inputs, ~7 MB, no copy-to-output rule — never read at runtime). The generated `Data/SeedData/g25_pca_population_samples.json` (~18k rows) + human-readable `g25_pca_seed_report.txt` sit alongside the distance seed; the JSON has its own `<None Update>` copy-to-output entry in `Odin.Api.csproj`.
- **Regenerating the JSON seed.** [`tools/GenerateG25PcaSeed`](tools/GenerateG25PcaSeed/Program.cs) (standalone console, **not** in `Odin.slnx`; references Odin.Api for the shared builder) reads the distance seed + the in-repo coordinate files and rewrites the JSON + report. No args needed now that the files are committed: `dotnet run --project tools/GenerateG25PcaSeed` (from `Odin/`); override with `--genetic-dir` if pointing at an external folder.
- **Applying it — three ways.** (1) Idempotent startup seeder `G25Seeder.SeedG25PcaPopulationSamplesAsync()` runs AFTER the distance seeder (so the eras exist) and short-circuits when the table is non-empty. (2) Admin re-import `POST /v1/api/admin/g25/import-pca-population-samples` (`AdminOnly`, `strict`, 5-min timeout) → `G25SeedImportService.ImportPcaPopulationSamplesAsync`. (3) On-demand from the repo files: unskip + run `G25PcaPopulationSamplesImportTests` (mirrors the export-test pattern — `ConnectionStrings__DefaultConnection` via user-secrets/env, derives from the committed files, binary-COPYs into `g25_pca_populations_samples`). All three are idempotent on the `(era, label, ids)` triple. **⚠️ Unlike the distance importer (skips by `Label`), the PCA existing-row key is the `(era, label, ids)` TRIPLE** — a PCA `Label` is NOT unique (many individuals per population), so a label-only skip would drop every individual after the first. All write paths (importer + CRUD create/update/delete) bust the per-era PCA scatter cache (see the next section); `UpdateAsync` busts BOTH the old and new era.
- **Tests.** `G25PcaSeedBuilderTests` (unit, fast) pins the matching rules on synthetic data; `G25PcaPopulationSamplesSeedTests` (unit, fast) validates the committed JSON artifact (25 numeric PCs per row, era id ∈ 1-6, every era present, Modern largest, no duplicate triples); `G25PcaPopulationSamplesImportTests` is the manual importer (`ConnectionStrings__DefaultConnection` via user-secrets/env → whatever DB that points at, so keep it pointed at the intended DB when running it).

## G25 per-era PCA scatter — owner-facing Ancient Origins "PCA" tab

The Ancient Origins → G25 "PCA" tab plots the user's own Global25 coordinate inside a 2D PCA scatter of that era's reference populations (from the seeded `g25_pca_populations_samples`). Enabled for **every** G25 order — it's the same coordinates the Distance tab uses, no paid gate.

- **Fitted PER era, server-side, cached.** [`G25PcaScatterService`](Odin.Api/Endpoints/G25PcaScatter/G25PcaScatterService.cs) loads an era's samples, mean-centers, and fits a fresh PCA ([`G25PcaEngine`](Odin.Api/Endpoints/G25PcaScatter/G25PcaEngine.cs) — 25×25 covariance → symmetric EVD via **`MathNet.Numerics`**; a new managed dep, no native bits) so that era's populations spread out on PC1/PC2. **G25 coords are already global PCs, but we refit per era for better within-era separation.** The FULL cloud is fit + projected; only the plotted `points` are **downsampled** (deterministic, stratified-by-population, ~2,500 cap) so Modern's ~10.9k stays fast. Per-population **centroids** are computed from the full projection. Cached per era (`G25SampleCacheKeys.PcaScatter(eraId)`, 1h TTL, skipped under `Testing`), invalidated on every PCA-sample write (importer + CRUD; Update busts old+new era).
- **⚠️ Deterministic sign convention.** Eigenvectors are sign-ambiguous; `G25PcaEngine` forces each axis's max-abs loading positive so the plot never mirrors between cold fits — and it's **folded into the returned `axisX`/`axisY`** so there's a single basis the client reuses. A near-degenerate top-2 (PC2≈PC3) is logged (a sign convention can't fix an axis swap). Sparse eras (< ~10 samples) return `basis: null` → the FE shows an info alert.
- **Endpoint:** `GET /v1/api/g25-pca/eras/{eraId:int}` ([`G25PcaScatterEndpoints`](Odin.Api/Endpoints/G25PcaScatter/G25PcaScatterEndpoints.cs), **`EmailVerified`** — the reference cloud is generic public reference data, mirroring `g25-distance-eras`; the user's coordinate never appears in the response). Returns `{ eraId, eraName, basis: { means[25], axisX[25], axisY[25] } | null, points[], centroids: [{ label, x, y, sampleCount, coordinates[25] }], totalSamples, plottedSamples }`. Each centroid carries its **25-dim mean** (`coordinates`) so the FE can rank clusters by genetic distance to the user (same metric as the Distance tab). `.Produces<T>()` + `.Produces(404)`.
- **User point is projected CLIENT-side** from the returned basis (`(coord − means)·axisX`, `·axisY`), so the response stays user-agnostic and cacheable. The `app_store_transactions` row is **KEPT** (`Refunded`) as the financial audit record. The job runs in the **background** (the webhook stays fast + returns 200), is **idempotent** (re-run no-ops), and **best-effort per asset** (an R2/tools-api failure is logged; the daily R2 + weekly merge orphan sweeps reclaim leftovers). Covered by `RefundCleanupJobTests` (full qpAdm purge + add-ons + R2; shared-file preserved + sibling survives; G25; idempotent).

New `*Contract` fields ⇒ run `gen:api` (done).

### Paid Y-DNA results unlock — server-enforced $9.99 paywall (per order)

The order's **Y-DNA clade is withheld from the API until a one-time $9.99 unlock is purchased** (iOS in-app purchase). This is **server-enforced**, not a client lock: the clade never leaves the backend until the entitlement exists.

> **⚠️ Y-DNA clade COMPUTE job deploy-resilience.** The clade is computed by `IYHaplogroupComputeService.ComputeAndPersistAsync` (Hangfire, enqueued at order creation + re-enqueued on result view when the `QpadmCladeResult` row is `null` or `Unavailable` — `OrderService.BuildYDnaResultAsync`). It writes the row only at a TERMINAL status (no persisted "in-progress" state), so a worker death / redeploy mid-run leaves no row → the next view re-enqueues it → self-heals. Its **`[AutomaticRetry(Attempts = 3, OnAttemptsExceeded = Fail)]` was on the CONCRETE method and silently ignored** (Hangfire reads filters off the enqueued **interface** → it had been falling back to the default 10 retries); it now lives on `IYHaplogroupComputeService` (same load-bearing rule as `IMergeJob`), pinned by `YHaplogroupComputeRetryTests`.

- **Gating (`OrderService.GetQpadmResultForOrderAsync` → `ApplyYDnaLockAsync`).** The `qpadm-result` response is built + cached with the **full** clade as before, but the lock is applied **per-request on a COPY** (never mutating the shared/cached instance) from the live entitlement: when the result is `Completed` with a real haplogroup (`YDna.Clade.Clade` non-empty) and there's **no** `QpadmYDnaUnlock` row for the order (and the caller isn't an admin), it returns a copy whose `YDna.Clade = null` + **`YDna.Locked = true`**. Per-request gating (not gating-into-the-cache) means a purchase takes effect immediately with **no cache invalidation**, and an admin's unlocked view can't poison the owner's cached (locked) view. Info states (NotApplicable/NoYData/Pending/…) and a Completed-but-couldn't-determine clade are **never** locked (nothing of value to sell). `Locked` is a new field on `GetOrderQpadmResultContract.YDnaResult`.
- **Purchase (`POST /v1/api/orders/{id}/ydna/purchase`, `EmailVerified`, `OrderEndpoints.PurchaseYDna` → `OrderService.PurchaseYDnaUnlockAsync`).** Validates the StoreKit add-on JWS via `IAppStorePurchaseService.ValidateAddOnTransaction(jws, AppleIap:YDnaProductId)` (= `io.ancestrify.app.ydna`), records a **`QpadmYDnaUnlock`** (`qpadm_ydna_unlocks`; unique `TransactionId` → anti-replay; `OrderId` indexed for the per-view lookup), and returns the now-unlocked `YDnaResult`. Ownership-checked (order's `CreatedBy`). Migration `AddQpadmYDnaUnlock`.
  - **⚠️ A transaction unlocks EXACTLY ONE order — reject a cross-order replay (load-bearing, was a paywall-bypass bug).** The Apple transaction carries no order binding, so the endpoint binds it to the first order it's redeemed against. A replay of the SAME transaction against the SAME order is idempotent (retry / app-kill); a replay against a **different** order the buyer owns is **rejected with 400** ("already used to unlock a different order") and returns **no clade**. Without this, one $9.99 buy could reveal every order the buyer owns by re-POSTing the same receipt to each order's endpoint (and the response would hand back that order's clade for free) — the unique `TransactionId` index stops a second *row* but not a second *reveal*, so the guard is in code (`existing.OrderId != orderId` check, plus a re-read in the unique-index race catch). Mirrors how `AncestralPortraitService.PurchaseAsync` returns the set bound to the original order, never the requested one.
  - **Refund/revoke.** The Y-DNA unlock lives in its own table (not `app_store_transactions`), so `AppStoreWebhookEndpoints` now **deletes the matching `qpadm_ydna_unlocks` row on `REFUND`/`REVOKE`** — the per-request gate reads the table live, so the clade re-locks immediately. (The AI-portraits add-on has the same not-yet-reconciled-on-refund gap in `ancestral_portrait_sets` — out of scope here; revoking it would mean deleting generated images.)
- **Tests:** `QpadmOrderCladeEndpointsTests` — `…IsLockedUntilPurchased` (Locked + no clade as the owner), `…AdminBypassesTheLock` (admins see clades), `PurchaseYDna_UnlocksTheCladePayload`, `…IsIdempotentOnTransactionId`, **`…CannotReplayTransactionAgainstADifferentOrder`** (the bypass guard). The paywall tests act as a **non-admin owner** (`CreateClientAsAsync(role: User)`) — the default test `Client` is Admin, which bypasses the lock. Add-on JWS via `BuildAppStoreTransactionJws(productId: "io.ancestrify.app.ydna")`.
- **Web note:** the web (odin-charmander) has no IAP, so a locked order shows the web Y-DNA panel without a clade. The app is pre-launch (no public web users); a web unlock path is out of scope here. Re-run `gen:api` for the new `Locked` field + the purchase endpoint.

> **The AI "Through the Ages" add-on is priced at $9.99 too** (was $7.99) — set the same tier in App Store Connect for `io.ancestrify.app.aiportraits`. It was already server-validated/purchase-gated (`AncestralPortraitService.PurchaseAsync`); only the price changed (the local `Ancestrify.storekit` test config now reads 9.99 for both `.aiportraits` and `.ydna`).

## OpenAI image generation — admin-only `gpt-image-2` endpoints (via the OpenAI .NET SDK)

`/v1/api/admin/images/*` ([`ImageGenerationEndpoints`](Odin.Api/Endpoints/ImageGenerationManagement/ImageGenerationEndpoints.cs), **all `AdminOnly`**) wrap OpenAI `gpt-image-2` for programmatic image generation, consumed by odin-charmander + odin-bulbasaur-ios. Surface: `POST /generate` (text-to-image), `POST /generate-from-references` (edits), `POST /reference-images` (multipart upload) + list/get/delete, `GET /jobs` + `GET /jobs/{jobId}` (poll) + `DELETE`, `GET/PUT /settings`, `GET /usage`.

- **Official `OpenAI` SDK, behind a fakeable seam.** Generation/edits use `OpenAI.Images.ImageClient`, wrapped by [`IOpenAIImageClient`](Odin.Api/Endpoints/ImageGenerationManagement/OpenAiClient/IOpenAIImageClient.cs) so integration tests substitute `FakeOpenAIImageClient` and **CI never calls OpenAI**. The usage/cost endpoints aren't in the SDK, so they're a small `HttpClient` call inside the same impl.
- **Two keys, isolated.** `OpenAI:ApiKey` (project key) does generation/edits ONLY; `OpenAI:AdminApiKey` (org Admin key) is used ONLY by `GET /usage` (Administration API). An Admin key can't generate and an image key can't read usage. Secrets via user-secrets locally / Coolify env in prod (`<secret>` placeholders in appsettings) — `dotnet user-secrets set "OpenAI:ApiKey" …`.
- **⚠️ `GET /usage` reads the COMPLETIONS usage endpoint, not `/usage/images`.** gpt-image-2 is a GPT model, so OpenAI bills/reports it under **completions** — `GET /v1/organization/usage/completions?models=gpt-image-2` (request + input/output token counts) — and the legacy `/v1/organization/usage/images` reads **0** for it (that endpoint only covers the old DALL-E image API). Costs come from `/v1/organization/costs` (org-wide, settle ~a day late). The endpoint **soft-fails**: a missing Admin key / upstream error sets `OpenAiError` on a 200 response (not a 4xx/5xx) so the admin page shows a clear note instead of erroring. The bucket `limit` is clamped per `bucket_width` (1d→31, 1h→168, 1m→1440); longer ranges page via `next_page`. The **odin-charmander** "Image Usage" admin page (`/dashboard/image-usage-admin`) renders this and **live-refreshes over SignalR** — the service broadcasts `ImageUsageChanged` (via `IImageGenerationRealtimeNotifier.NotifyUsageChangedAsync`, `Clients.All`) whenever a job reaches a terminal state; the FE invalidates the `["admin","images","usage"]` query.
- **⚠️ gpt-image-2 SDK gotchas (in [`OpenAIImageClient`](Odin.Api/Endpoints/ImageGenerationManagement/OpenAiClient/OpenAIImageClient.cs)).** (1) **Never set `ImageGenerationOptions.ResponseFormat`** — gpt-image-2 rejects `response_format` and always returns base64 (the SDK fills `GeneratedImage.ImageBytes`). (2) **`GeneratedImageQuality.High` serialises to DALL-E's `"hd"`** — construct quality/size/background/format/moderation from the canonical strings (`new GeneratedImageQuality("high")`), never the named statics. (3) **Multi-image edits go through the SDK protocol method** `GenerateImageEditsAsync(BinaryContent, contentType, RequestOptions)` with a hand-built `image[]` multipart — the typed edit overloads accept only one input image. (4) The gpt-image-2 options are `[Experimental("OPENAI001")]`; the diagnostic is `#pragma`-suppressed in that one file. (5) **gpt-image-2 needs a verified OpenAI org** — a 403 / "model not available" in prod is account state, not a code bug.
- **Sync default, async opt-in.** `POST /generate` runs inline and returns 200 with the images; `async: true` enqueues [`IImageGenerationWorker`](Odin.Api/Endpoints/ImageGenerationManagement/IImageGenerationWorker.cs) (Hangfire, **`default` queue**) and returns 202 + a job id to poll, with a SignalR `ImageJobChanged` push to the initiating admin (`Clients.User(sub)`, [`ImageGenerationRealtimeNotifier`](Odin.Api/Hubs/ImageGenerationRealtimeNotifier.cs)). **`[Queue]`/`[AutomaticRetry]` live on the worker INTERFACE** (same load-bearing rule as `IMergeJob`; asserted by `ImageGenerationWorkerTests`). A terminal failure (moderation/bad request → recorded `Failed`) is not retried; transient 429/5xx rethrows for Hangfire retry; a dead worker is reconciled to `Failed` by `ImageGenerationJobFailureStateFilter`. `moderation_blocked` surfaces as **422**, 429 as 429, other upstream errors as 502.
  - **⚠️ Deploy-interruption self-heal (same fix as the portraits job).** `ProcessJobAsync` **atomically CLAIMs** the job (Pending/Failed → Running, or re-claims a **stale** Running whose `UpdatedAt` is older than `StaleJobMinutes = 20`); a fresh Running job is skipped (`claimed == 0`) so a duplicate enqueue / the reconcile racing the live worker never double-generates. No per-call heartbeat is needed — a single OpenAI generate/edit call is bounded (~1–2 min even at "high" quality), so a live run is never 20 min old. A recurring **`image-generation-reconcile-stale`** job (`*/5 * * * *`, `IImageGenerationWorker.ReconcileStaleJobsAsync`) re-enqueues any job stuck Running past the window, so a redeploy mid-run recovers in minutes instead of waiting on Hangfire's slow retry/timeout path (the `FailureStateFilter` remains the terminal backstop). On a failed run, this run's uploaded R2 objects are deleted + the poisoned `GeneratedImage` inserts detached, so an interrupted run leaves no orphans (keys are deterministic per `(jobId, index)`, so a clean re-run also overwrites leftovers). Guarded by `ImageGenerationSelfHealTests` (unit) + `ImageGenerationClaimTests` (integration/Postgres).
- **Persistence.** Every request is an `image_generation_jobs` row (Guid PK = job id = SignalR correlation) with child `generated_images`; uploads are `reference_images`; defaults are the single-row `image_generation_settings` (dedicated entity, memory-cached, seeded on first read — NOT the bool-only `AppSettings`). Image bytes live in R2 (`openai/images/{jobId}/{i}.{ext}`, `openai/reference-images/{guid}.{ext}`); Postgres holds metadata + persisted OpenAI token `usage` (first-party cost tracking that doesn't depend on the lagging Administration API). `IR2Storage` gained `DownloadAsync` so edits can stream reference bytes back to OpenAI. Guardrails (max n, allowed qualities/formats/backgrounds, upload size, max total pixels) are `ImageGenerationLimitsOptions`; reference uploads are magic-byte validated (`FileSignatureValidator.IsSupportedImage`, PNG/JPEG/WEBP only).
- **Selectable dimensions + quality.** Callers pick `size` and `quality` (and background/format/etc.) per request; omitted values fall back to the editable `image_generation_settings` defaults. gpt-image-2 accepts **arbitrary custom dimensions**, so `size` is NOT a fixed allow-list — `ImageParameterVocabulary.ValidateSize` enforces the model rules (`auto`, or `WxH` with both edges multiples of 16, ≤3840px, ≤3:1 aspect, total pixels 655,360–8,294,400); `ImageGenerationLimitsOptions.MaxImagePixels` adds a configurable cost cap. Quality is `auto`/`low`/`medium`/`high` (`high` runs a 4-stage pipeline ~30–50× slower than `low`).
- **Tests.** `CustomWebApplicationFactory` swaps `IOpenAIImageClient` → `FakeOpenAIImageClient` (sentinel prompts: `moderation-boom` → 422, `rate-boom` → 429) and `IR2Storage` → in-memory `FakeR2Storage`. Coverage: `ImageGenerationEndpointsTests` (integration) + `FileSignatureValidatorImageTests` / `ImageGenerationWorkerTests` (unit). New `*Contract`s mean the FE schema is stale until `gen:api` is re-run; iOS adds the methods to `OdinAPI.swift` by hand.

## Per-user face-photo set — `gpt-image`-bound biometric capture from the iOS app

`/v1/api/users/face-photos` ([`UserFacePhotoEndpoints`](Odin.Api/Endpoints/UserFacePhotoManagement/UserFacePhotoEndpoints.cs), all `EmailVerified`) stores a per-USER (identity-scoped) set of face photos the iOS app captures via a guided ARKit multi-angle scan, so a **future** AI-image pipeline can generate images of the user. This is a **deliberate new endpoint group** (the iOS "no backend changes" rule's second sanctioned exception after IAP — the feature's value requires server persistence). Bytes live in **R2** (private), metadata in `user_face_photos`. Surface: `POST /` (multipart replace-set), `GET /` (caller's set metadata), `GET /{id}/download` (authenticated bytes), `DELETE /{id}` (one), `DELETE /` (clear all).

- **Identity-scoped, not order-scoped.** Unlike the profile picture (an order-level DB blob on `GeneticInspection`), this hangs off `User` (`UserFacePhoto.UserId` FK, resolved via `Users.RequireByIdentityAsync`). Every read/write filters by the caller's `UserId`; `download`/`delete-one` load by id then assert `row.UserId == user.Id` (else 403), mirroring `GetProfilePictureAsync`.
- **⚠️ Private, never a public R2 URL (biometric data).** Do NOT call `GetPublicUrl` for face photos — `R2Storage.UploadAsync` stamps `Cache-Control: immutable` and the public domain serves anyone with the URL, which is wrong for face data and impossible to revoke. The list returns metadata + an **authenticated** relative `downloadUrl` (`/v1/api/users/face-photos/{id}/download`) streamed via `Results.File`. The future AI pipeline reads bytes server-side with `r2Storage.DownloadAsync(key)` — no public URL is ever needed. (This is the one place that diverges from the admin-only `ReferenceImage`, which exposes `Url`.)
- **Replace-set semantics = idempotency.** `POST` REPLACES the caller's whole set (not append): upload new bytes to R2 → swap rows in one `SaveChanges` → THEN best-effort `DeleteAsync` the old R2 keys (commit-DB-before-deleting-old-bytes, so a crash never leaves the DB pointing at absent bytes; orphaned R2 objects are the lesser, sweepable evil). A retried/duplicate batch therefore converges instead of doubling. SHA-256 dedupe within the batch is the backstop. R2 cleanup on every delete path is **explicit** (EF cascade drops rows, never R2 objects).
- **Validation / limits (`UserFacePhotoLimits`).** ≤20 files, ≤2.5 MB each, ≤40 MB total (stays under the global 50 MB request cap — a naive 20×5 MB would 413). JPEG only, authoritatively checked with `FileSignatureValidator.IsJpeg` (client content-type never trusted). The iOS app normalizes each frame to ~1536 px, so real sets are far under these caps. Width/Height are read best-effort from the JPEG SOF marker (no image lib; (0,0) on parse failure — informational only). R2 key: `users/{sanitize(identityId)}/face-photos/{guid:N}.jpg` (identity prefix enables the per-user orphan-cleanup sweep — see "R2 orphan cleanup" below).
- **Wiring.** `DbSet<UserFacePhoto>` (config auto-discovered), `services.AddScoped<IUserFacePhotoService, …>`, `v1.MapUserFacePhotoEndpoints()`. Migration `AddUserFacePhotos` (table `user_face_photos`, FK → `application_users`, indexes on `UserId` and `(UserId, Sha256)`). New `*Contract`s ⇒ run `gen:api`; the iOS client adds the methods to `OdinAPI.swift` by hand (`uploadFacePhotos`/`facePhotos`/`facePhotoData`/`deleteFacePhoto`/`clearFacePhotos`).

## R2 orphan cleanup — delete storage for users no longer in the DB

All PER-USER private R2 data lives under **`users/{slug}/…`** where `slug = UserStorageKeys.Sanitize(IdentityId)` (the **one** shared slug helper — `Storage/UserStorageKeys.cs`; the two upload services delegate `SanitizeIdentity` to it so the sweep can't drift). Two trees today: `users/{slug}/face-photos/…` and `users/{slug}/ancestral-portraits/…`. When a user is deleted the DB rows cascade away, but the R2 objects (biometric face images) would linger — so a reconciliation sweep reclaims them.

- **`R2OrphanCleanupService`** (`Endpoints/StorageManagement/`): LIST the `users/{slug}/` common-prefixes in R2 → compute the set of `Sanitize(IdentityId)` for **all** current DB users → any R2 slug **not** in that live set is an orphan; delete its whole `users/{slug}/` tree. **Safe-by-direction invariant:** `Sanitize` is many→one, so a slug shared with ANY live user is kept — the sweep only ever deletes a slug **no** live user maps to, so a real user's data can never be removed. **Two circuit breakers** (the failure mode is asymmetric — under-counting R2 just deletes less, but under-counting the LIVE set would wipe real biometric data): it ABORTS without deleting if (1) **0 live users but R2 has data** (a wrong/empty DB connection — never "all users deleted"), or (2) **>50% of slugs orphaned** in one sweep once there are ≥20 slugs (the live set looks wrong — e.g. a partial restore). Each orphan slug deletes **best-effort** (one R2 hiccup logs + continues; next day retries). `RunAsync(dryRun)` previews non-destructively; returns an `R2CleanupReport` (scanned/live/orphans/deleted + `Aborted`/`AbortReason`).
- **`IR2Storage` gained** `ListKeysAsync` (paginated all-keys-under-prefix), `ListCommonPrefixesAsync` (delimited "sub-folders"), `DeleteManyAsync` (per-object idempotent loop — R2's multi-object delete has checksum quirks; orphan volumes are tiny).
- **Triggers:** a daily Hangfire recurring job **`r2-orphan-cleanup`** (`0 4 * * *` UTC, `SweepAsync` = real delete) + an AdminOnly **`POST /v1/api/admin/storage/cleanup-orphans?dryRun=`** (defaults to **dry-run** so an accidental call can't wipe data; pass `dryRun=false` to delete). `[AutomaticRetry(Attempts = 0)]` is on the **interface** method (Hangfire reads filters off the invoked interface — see the merge-jobs rule). Guarded by `R2OrphanCleanupServiceTests` (orphan deleted, live kept, dry-run no-op, special-char identity kept).

## Paid "Through the Ages" AI ancestral portraits — gpt-image-2 from the user's face photos

`/v1/api/orders/{orderId}/ancestral-portraits` (list) + `/v1/api/ancestral-portraits/sets/{setId}/*` + `/v1/api/ancestral-portraits/{id}/*` ([`AncestralPortraitEndpoints`](Odin.Api/Endpoints/AncestralPortraitManagement/AncestralPortraitEndpoints.cs), all `EmailVerified`) is a **paid add-on** that reimagines the user as **EVERY ancestral population in EVERY era** of their qpAdm result (one portrait per population per era — NOT just the era's top source), generated by **`gpt-image-2` edits** with the user's stored **face photos as references**. Outputs are a shareable carousel SET + a vertical reel (both iOS). The actual gen is reused, not rebuilt — only the reference source + a per-user gate are new.

- **Re-buyable iterations (each purchase = a new kept set).** An order holds **MANY** sets — the user can buy "Ages" again and again, and every iteration is kept + listed. Endpoints: **`GET /orders/{orderId}/ancestral-portraits`** → ALL iterations newest-first (`ListSetsAsync`); **`POST …/purchase`** → creates a NEW set each genuine transaction (a *replayed* transaction is still idempotent via the unique `TransactionId`); **`GET /ancestral-portraits/sets/{setId}`** → poll one iteration; **`POST /ancestral-portraits/sets/{setId}/generate`** → retry one iteration; **`DELETE /ancestral-portraits/sets/{setId}`** → `DeleteSetAsync` removes the iteration's **PRIVATE R2 portrait objects** (these are images of the user's face — never orphan them) THEN cascade-deletes the rows, ownership-checked; **returns 409 if the set is mid-generation** (a fresh `Running` — don't tear a set out from under the worker; a *stale* Running is deletable). (The old single-set-per-order `GET`/`generate-by-orderId` are gone — `RequestGenerateAsync` now takes a `setId`.) `PurchaseAsync` also catches the concurrent same-transaction insert race (unique `TransactionId`) and returns the winner's set (200) instead of a 500.
- **Multi-select (toggle).** `POST /ancestral-portraits/{id}/select` **TOGGLES** that variation's `IsSelected` — the user can keep any subset per era, all of them, or none (no more "exactly one per era"). The default after generation is variation 0 of each era pre-selected.
- **No duplicate portraits (load-bearing).** A doubled-portraits bug came from the generation job running twice (duplicate enqueue / concurrent worker). `RunGenerationAsync` now **atomically claims** the run with a conditional `ExecuteUpdateAsync` (→ `Running` only when not `Succeeded` and not a *fresh* `Running`; a **stale** `Running` set — `UpdatedAt` older than `StaleRunMinutes` = 10, i.e. a worker that died mid-run — IS re-claimable so a crashed generation self-heals instead of being stuck "Painting…" forever; `RequestGenerateAsync` likewise re-enqueues a stale Running set). Only one worker wins the claim → no double-run. Belt-and-braces: era groups are **de-duped by `EraId`** before generating, and `MapSet` collapses any duplicate variations (`GroupBy VariationIndex`) so already-affected data also renders clean.
- **⚠️ Stuck "Painting…" after a redeploy — orphaned-run SELF-HEAL (captured bug).** A backend **redeploy/restart mid-generation** killed the worker before the single final `SaveChanges`, leaving the set `Running` with **0 committed portraits**; Hangfire re-fetched the job but the second worker **no-op'd** because the orphaned set's `UpdatedAt` was only ~6 min old (< the old 30-min stale window), so it stayed stranded forever with nothing to re-trigger it (the iOS client only polls). Three-part fix: **(1) heartbeat** — `RunGenerationAsync` does a tiny `ExecuteUpdate` on `UpdatedAt` after **every population**, so a live multi-minute run keeps `UpdatedAt` fresh and is never mistaken for a dead one (a direct UPDATE — it does **not** flush the pending all-or-nothing portrait inserts). **(2) `StaleRunMinutes` 30 → 10** — safe now that a live run heartbeats (its `UpdatedAt` is never >~1 min old), so a dead run goes stale within ~10 min. **(3) recurring `ancestral-portrait-reconcile-stale`** (`*/5 * * * *`, `IAncestralPortraitWorker.ReconcileStaleRunsAsync`) re-enqueues any set still `Running` past the window → the claim re-claims the now-stale set and regenerates; bounded (a re-run ends `Succeeded`/`Failed`, never re-loops). **Don't** raise `StaleRunMinutes` back up without keeping the heartbeat, and keep `[Queue("default")]`/`[AutomaticRetry(0)]` on the worker **interface** for `ReconcileStaleRunsAsync` too. Guarded by `AncestralPortraitSelfHealTests` (unit: interface filters + reconcile re-enqueues only stale Running) and `AncestralPortraitGenerationClaimTests` (integration/Postgres: fresh Running not reclaimed, stale Running reclaimed).
- **No orphaned R2 (biometric) objects.** R2 uploads happen per-image inside the loop but the rows commit only at the single final all-or-nothing `SaveChanges`; so on ANY throw before that commit the run's uploads would be orphaned. The catch now **deletes every key uploaded this run** (`uploadedKeys`) and detaches the poisoned tracked inserts so `FailAsync` cleanly persists `Failed`. (A re-run reuses the same deterministic keys, overwriting any leftovers.)

- **One portrait per population in every era (load-bearing — this is the headline behavior).** `RunGenerationAsync` flattens the qpAdm result to one **(era, population)** target per population in every era — ranked by ancestry % within the era and capped at **`MaxPopulationsPerEra`** (a cost rail; default 8 ⇒ all in practice) — and generates a portrait group for each. It is NOT just the era's top source. Each portrait row carries `PopulationId` + `PopulationName`; the R2 key includes the population (`…/{eraId}-{populationId}-{i}.{ext}`) so groups can't collide; `produced/targets.Count` accounting + `set.Error` are over (era,population) pairs. `MapSet` groups portraits by **(EraId, PopulationId)** (ordered era → ancestry-rank) and the contract `AncestralPortraitEraContract` carries `PopulationId` (the list field is still named `Eras`, but each entry is an era×population group). Migration `AddPopulationToAncestralPortraits` (adds `population_id` + the `(SetId, EraId, PopulationId)` index; settings `MaxPopulationsPerEra` defaults to **8** in the migration so the existing singleton row isn't 0 → Take(0) → nothing).
- **Reuses the admin image pipeline at the SDK seam.** `AncestralPortraitService.RunGenerationAsync` builds a prompt per population (`AncestralPortraitPrompts.Build` — identity-preserving lead "the SAME person … reimagined as <population>" + the curated `QpadmPopulation.ImagePrompt` or a name+description fallback + a photoreal style) and calls `IOpenAIImageClient.EditAsync` directly with the caller's `UserFacePhoto` bytes (`r2Storage.DownloadAsync`, ≤`MaxFaceReferences`) as `OpenAIReferenceImage`s — the SAME edit path `ImageGenerationService.RunEditAsync` uses, but the refs come from the user's face set (not admin `ReferenceImage`s) and there's no admin gate. Runs on a Hangfire worker (`IAncestralPortraitWorker`, `[Queue("default")]`/`[AutomaticRetry(Attempts=0)]` on the **interface**); the iOS client **polls** (no SignalR). **The iOS app now uploads a SINGLE clear front-facing selfie** (not a multi-angle set), so generation typically runs from one reference photo — `MaxFaceReferences` is just a cap. **Output is a PORTRAIT:** the default size is `1024x1536` (vertical 2:3) and the prompt explicitly requests a "VERTICAL portrait-orientation, head-and-shoulders portrait … single subject centered and facing the camera". **Gender-consistent presentation:** `Build(…, Gender? gender)` adds a clause forcing **feminine** dress/hair/adornments for `Female` and **masculine** for `Male` (the client's gender comes from `order.GeneticInspection.Gender`), so a woman is rendered as a woman and a man as a man, not just "a person of the population". (`AncestralPortraitPromptsTests` asserts the lead, the gender clauses, and prompt length < 2200.)
- **Entities (migration `AddAncestralPortraits`; index relaxed in `AllowMultipleAncestralPortraitSetsPerOrder`).** `AncestralPortraitSet` (purchase `TransactionId` unique → anti-replay; **`OrderId` index is NON-unique → many iterations per order**; `Status`, `Error`, `CreatedAt` orders the history) + child `AncestralPortrait`s (per era × population × variation; `IsSelected` toggled per portrait — any subset). Bytes are **PRIVATE** in R2 (`users/{identityId}/ancestral-portraits/{setId}/{eraId}-{populationId}-{i}.{ext}`, ext from the configured output format), served only via the authenticated `GET /ancestral-portraits/{id}/download` (`Results.File`) with an ownership check — **never `GetPublicUrl`** (images of the user's face). Delete/regenerate explicitly remove the R2 objects.
- **Add-on purchase (payment-critical).** The add-on isn't a qpAdm/G25 `ServiceType`, so `IAppStorePurchaseService` gained **`ValidateAddOnTransaction(jws, expectedProductId)`** (same signature/chain/bundle/env checks as `ValidateTransaction`, but matches `productId` to `AppleIap:AiPortraitsProductId` = `io.ancestrify.app.aiportraits` instead of the product↔service map). `PurchaseAsync` validates it, records the set (unique `TransactionId`), and enqueues generation. Unit-tested in `AppStorePurchaseServiceTests` (add-on cases) + `AncestralPortraitPromptsTests`. **Needs a real Sandbox + crafted-JWS integration test + the App Store Connect product before shipping.**
- **Generation is fully runtime-configurable from the web admin (migration `AddAncestralPortraitSettings`).** The model, size, quality, background, output format, moderation, `VariationsPerEra`, `MaxEras`, **`MaxPopulationsPerEra`**, `MaxFaceReferences`, and the cost-estimate rates are a **single-row DB entity** `AncestralPortraitSettings` (`Id = 1`, `ancestral_portrait_settings`) behind `IAncestralPortraitSettingsService` — **the exact `ImageGenerationSettings`/`ImageSettingsService` pattern** (memory-cached, 5-min TTL, `UseCache => !Testing`, seeded from the entity's property defaults on first read, cache invalidated on write). The worker reads it per run via `GetAsync` (no redeploy to retune; **change the model from `gpt-image-2` here too**). **`VariationsPerEra` defaults to 1** — ONE image per (era, population) group (raise it for variations to pick between); **`MaxPopulationsPerEra` defaults to 8** — a portrait is generated for EVERY population in EVERY era up to this cap (ranked by ancestry %), the cost rail for the per-population fan-out. Edit via **`GET`/`PUT /v1/api/admin/ancestral-portraits/settings`** (`AdminOnly`; `Request` validates against the shared `ImageParameterVocabulary` + `ImageRequestValidation.ValidateEnum`, caps `VariationsPerEra` 1–8 / `MaxEras` 1–20 / `MaxPopulationsPerEra` 1–20 / `MaxFaceReferences` 1–16, cost rates ≥ 0). The **old `AncestralPortraitLimitsOptions` is deleted** — its fields moved into this entity. The output format drives the R2 key extension + `ContentType` (`ResolveContentType` → jpeg/png/webp). Per-population upstream failures are caught (the set still succeeds with the populations that worked); `moderation_blocked` → the set's `Error`. Prompts are population-derived (no free-text user input) → low abuse/moderation risk.
- **Cost/usage tracking (migration `AddAncestralPortraitUsage`).** Each run sums the gpt-image-2 `usage` across the per-era edits onto `AncestralPortraitSet` (`UsageInputTokens`/`OutputTokens`/`TotalTokens`, `ImageCount`, `EstimatedCostUsd` from `AncestralPortraitSettings.CostPerMillion{Input,Output}TokensUsd` — admin-tunable to OpenAI's rates) and **logs it** at completion. **It is deliberately NOT exposed to the iOS app/user** — not in `AncestralPortraitSetContract.Response`, not in any client field. Cost is surfaced ONLY via `GET /v1/api/admin/ancestral-portraits/usage` (**`AdminOnly`** — total runs/images/tokens/estimated USD), consumed by the **web** admin app. This first-party tally is real-time (unlike OpenAI's lagging usage API). Also: `input_fidelity` is **never** sent — gpt-image-2 rejects it (gpt-image-1-only).
- **Wiring.** `DbSet`s (incl. `AncestralPortraitSettings`), `services.AddScoped<IAncestralPortraitService/SettingsService/Worker>`, `v1.MapAncestralPortraitEndpoints()`. **Needs `OpenAI:ApiKey` (already used by the admin image gen) + run `gen:api`** (new `/admin/ancestral-portraits/settings` GET/PUT + usage contracts for the web admin UI).
- **Per-population creative — GENDER-SPECIFIC built-in scenes (so the portrait resembles the population's own photo).** `AncestralPortraitPrompts.Build` resolves the per-population scene in priority order: (1) the admin override `QpadmPopulation.ImagePrompt`; else (2) a built-in **gender-specific** scene from [`AncestralPortraitScenes`](Odin.Api/Endpoints/AncestralPortraitManagement/AncestralPortraitScenes.cs) — a `Name → (Male, Female)` map covering all 34 seeded populations, selected by `order.GeneticInspection.Gender` (Female → women's variant, Male/unknown → men's); else (3) the bare name+description fallback. The men's variants are adapted from the FE manifest `POPULATION_IMAGE_PROMPTS` (which still drive the male-only population AVATARS); the **women's variants are independently period-researched** (documented garments/headwear/hairstyle/jewellery per culture — not a recoloured men's outfit, no men's armour unless attested) and were generated via a Workflow (one agent per population: author male+female → adversarial verify/fix), each scene ≤ 800 chars. Identity + the gender clause still come from `Build`'s wrapper. The keys must match `Data/SeedData/qpadm-populations.json` names exactly — a name-format drift silently falls through to (3).
- **⚠️ Identity rule (load-bearing) — keep face/skin/eyes; the hair/beard directive is PER-POPULATION and SELF-CONTAINED (no shared male clause).** The wrapper has four parts: (1) `IdentityLead` keeps the user's facial features, bone structure, **natural skin tone/pigmentation and eye colour** ("never lighten, darken or recolour the skin or eyes") — note it deliberately does **not** say "likeness" (gpt-image-2 reads that as "keep the hair too"); (2) the per-population subject scene **plus** `AppendHairDescription`; (3) `GenderClause` = the culture's gendered attire; (4) the **women-only** `FemaleHairEnforcement` (restyle the hair, keep her clean-faced); (5) `Style`. gpt-image-2 copies the selfie's hair unless given BOTH a concrete described target AND the negation — so each population states both, **in its own descriptor**:
  - **Men: each `MaleHair` is a complete, self-contained instruction** appended verbatim by `AppendHairDescription` (no `His period hair and beard:` label, no shared enforcement clause). The default form is `"Give him <period hair + beard>, replacing the hair and facial hair from the reference photo (never keep the photo's own)"`. **There is intentionally NO global/shared male hair-enforcement clause** — the previous `HairEnforcement` male/unknown branches were removed. This was a deliberate product call: **prompts must be unique per population and each must fully own its own hair/beard behaviour**, so a population can diverge. The headline divergence: **`Anatolian Neolithic Farmer` KEEPS the client's own facial hair** (`"…but keep his own facial hair from the reference photo exactly as it is - never add, remove or restyle his beard or moustache"`) — only its hair restyles; all other 33 males replace the photo's hair+beard. To change one population's beard behaviour, edit ONLY that population's `MaleHair` string.
  - **Women: unchanged** — `AppendHairDescription` appends `Her period hairstyle: <FemaleHair>.` and the small shared `FemaleHairEnforcement` forces the restyle + clean-faced look. (The `FemaleHair` descriptors are still period-researched per culture.)
  - **Still committed, not generic:** a male `MaleHair` must still describe a DISTINCTIVE, COMMITTED period look (a generic/optional "clean-shaven OR a beard" lets the model keep an already-bearded selfie). Don't reintroduce a shared male clause, don't reintroduce skin/eye changes, and keep each male descriptor's replace-or-keep directive explicit.
  Fallback description capped at **800 chars**; assembled prompt stays under the **~2400-char rail** (no runtime limit — quality guard; removing the shared male clause made male prompts shorter). Guarded by `AncestralPortraitPromptsTests` (`Build_UsesBuiltInGenderedScene_…`, `Build_AdminOverrideBeatsBuiltInScene`, **`Build_PreservesSkinAndEyeColour_ButReplacesHairAndBeard_ForMaleViaSelfContainedDirective`**, **`Build_KeepsClientsOwnFacialHair_ForAnatolianNeolithicMale`**, `Build_StaysUnderLengthRail_ForLongestSeededScene`).
